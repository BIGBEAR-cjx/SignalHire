# 搜索研究闭环 V1 设计

## 目标

把 SignalHire 的“搜人”从一次性查询升级成可连续迭代的人才研究工作流。第一阶段聚焦一条闭环：用户编辑搜索条件，手动启动搜索，实时看到搜索过程，审阅候选人和证据，用选择题反馈结果质量，再生成下一轮优化搜索条件。

## 用户场景

主要用户是公司 HR 和猎头。用户通常围绕一个招聘项目持续寻找 AI 方向人才，而不是只做一次搜索。用户需要知道系统为什么这样搜、搜了哪些来源、哪些证据足够强、哪些缺口需要补搜，以及下一轮应该如何调整条件。

## 当前基础

代码中已经有这些能力雏形：

- `ResearchTool` 支持项目上下文、手动搜索、停止搜索、轮询队列状态、实时 feed、结果展示和下一轮反馈搜索。
- `ResearchTimelinePanel` 已展示搜索过程和状态。
- `EditableSearchPlanPanel` 已允许从输入生成可编辑搜索计划，再手动启动。
- `SEARCH_FEEDBACK_GROUPS` 已提供选择题反馈。
- `buildFeedbackOptimizedSearchInput` 已能把反馈转成下一轮搜索输入。
- 结果组件已经有搜索计划、来源执行、缺口补搜、证据覆盖、人才地图、候选人对比和候选人证据画像。

第一阶段不推翻这些基础，而是把它们重新组织成用户能理解的闭环。

## 范围

### 1. 搜索过程面板

搜索进行中时，用户应看到“系统正在做什么”，而不是只看到搜索计数。面板分为四块：

- 当前阶段：排队、拆解画像、搜索来源、读取网页、整理证据、生成候选名单、失败或停止。
- 搜索内容：最近搜索词和抓取链接，完整换行展示。
- 研究覆盖：基于当前 feed 推断已触达的来源类型，例如 GitHub、论文、公司页、公开资料、其他网页。
- 下一步提示：如果任务排队太久、状态服务不稳定、worker 离线或用户停止搜索，明确告诉用户可以等待、停止或调整条件。

实现上继续使用现有 `feed`、`live`、`jobStatus` 和 `buildResearchProgressView`，不新增后端事件协议。第一版只做前端推断和展示重组。

### 2. 反馈优化预览

用户完成结果审阅后，反馈区不只是一个“重新搜索”按钮，而是一个“下一轮搜索优化器”：

- 用户选择结果是否精准。
- 用户选择推荐人选是否满意。
- 用户选择主要问题。
- 用户选择下一轮优化方向。
- 系统展示将如何调整下一轮搜索：
  - 收紧或放宽匹配条件。
  - 扩展哪些来源。
  - 强化哪些证据要求。
  - 避免哪些不合适候选画像。

点击主按钮后，仍使用 `buildFeedbackOptimizedSearchInput` 生成下一轮输入并调用现有 `run()`。第一版不新增数据库表，也不持久化反馈。

### 3. 项目页下一步建议

项目详情页应从“候选人列表 + 历史搜索”向“项目研究空间”靠近。第一阶段只做轻量增强：

- 如果项目已有历史搜索，突出最近一次搜索。
- 如果候选池为空，提示先在本项目下启动搜人。
- 如果候选池存在候选人，提示优先审阅候选人对比和证据画像。
- 不新增复杂项目分析 API。

### 4. 双语

新增界面文案必须接入现有 `useI18n` 字典。搜索过程、反馈优化预览和项目页下一步建议至少支持中文和英文。模型生成内容、候选人姓名、来源 URL、论文名和公司名保留原文。

## 不做

- 不做导出功能。
- 不做新的候选人详情路由。
- 不做复杂 feedback 数据表。
- 不做团队协作和权限分层。
- 不做历史报告自动翻译。
- 不重写 worker 事件协议。
- 不大范围重构项目详情页所有旧中文文案。

## 组件设计

### `ResearchProcessPanel`

职责：把当前搜索状态和 feed 组织成用户可读的搜索过程。

输入：

- `loading`
- `jobStatus`
- `progressView`
- `feed`
- `live`
- `onStop`
- `locale`

输出：

- 当前阶段标题和说明。
- 搜索/抓取计数。
- 最近搜索内容列表。
- 来源覆盖 chips。
- 停止搜索按钮。

### `FeedbackOptimizationPreview`

职责：根据选择题反馈展示下一轮优化预览，并触发下一轮搜索。

输入：

- `feedback`
- `hasCoreSearchFeedback`
- `onChange`
- `onRun`
- `loading`
- `locale`

输出：

- 选择题控件。
- 优化预览列表。
- 主按钮“按反馈优化下一轮搜索”。
- 未满足核心反馈时的提示。

### `ProjectNextStepPanel`

职责：在项目详情页给用户一个明确下一步。

输入：

- 项目基础信息。
- 候选人数量。
- 历史搜索数量。
- 当前筛选状态。

输出：

- 下一步建议标题。
- 1 到 3 条行动建议。
- 搜人/核验入口。

## 数据流

1. 用户在项目或搜人页输入画像。
2. 用户可先生成可编辑搜索计划。
3. 用户手动启动搜索。
4. `ResearchTool.run()` 调用 `/api/search`，然后通过 NDJSON 或队列轮询更新 `feed`、`live` 和 `jobStatus`。
5. `ResearchProcessPanel` 使用这些状态展示实时搜索过程。
6. 搜索完成后展示结果组件。
7. 用户填写反馈。
8. `FeedbackOptimizationPreview` 展示下一轮优化预览。
9. 用户点击主按钮。
10. `buildFeedbackOptimizedSearchInput` 生成下一轮输入，复用 `run()` 启动下一轮搜索。

## 错误和边界

- 没有输入时不允许启动搜索。
- 搜索运行中可以停止。
- worker 排队过久时显示“研究服务繁忙”而不是空转。
- 状态服务查询失败三次后显示重连提示。
- 反馈没有选择“精准度”和“满意度”前，不允许启动反馈优化搜索。
- 反馈优化预览必须在没有结果时隐藏。
- 英文模式下新增固定文案必须显示英文。

## 测试策略

### 单元测试

新增 `research-loop.test.mjs`，覆盖纯函数：

- 从 feed 推断来源覆盖。
- 从 jobStatus/feed/live 推断搜索阶段。
- 从反馈状态生成优化预览文案 key。
- 缺少核心反馈时返回不可运行状态。

这些逻辑应放在 `web/lib/research-loop.mjs`，避免在 React 组件里难测。

### 构建验证

- `node --test research-loop.test.mjs`
- `node --test i18n.test.mjs`
- `npm run build`

### 手动验证

- 打开 `http://localhost:3000/app/search`。
- 输入 AI 人才画像，生成可编辑搜索计划，确认不会自动开始搜索。
- 点击启动搜索，确认搜索过程面板可见并可停止。
- 使用已有示例或完成结果后，选择反馈项，确认出现下一轮优化预览。
- 点击“按反馈优化下一轮搜索”，确认输入和搜索流程进入下一轮。

## 验收标准

- 用户在搜索运行中能看到明确的搜索阶段、搜索内容和来源覆盖。
- 用户能用选择题反馈结果是否精准、是否满意、主要问题和下一轮方向。
- 系统能在运行下一轮前展示优化预览。
- 点击反馈优化后能复用现有搜索流程发起下一轮搜索。
- 项目页能给出轻量下一步建议。
- 新增文案支持中英双语。
- 构建和相关单元测试通过。
