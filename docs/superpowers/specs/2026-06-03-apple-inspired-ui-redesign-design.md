# Apple 风格 UI 与交互重构设计

## 背景

SignalHire 的产品目标是为公司 HR 和猎头提供全球 AI 人才搜索平台：从论文、实践、开源、工作经历等多角度综合搜索并交叉验证候选人。当前功能已经覆盖项目、搜人、核验、候选池、历史、反馈和实时搜索进度，但界面仍偏传统后台工具：深色左侧栏、emoji 图标、灰底卡片和分散动作较多。它能用，但还没有形成高端、克制、可信的产品体验。

本次重构参考 Apple 官网的设计风格和美学，但不复制 Apple 的品牌、图形、文案或产品页结构。可借鉴的是设计原则：强焦点、少解释文字、轻导航、大留白、清晰主动作、材质感面板、渐进展示和精细动效。

## 目标

1. 把 SignalHire 从“功能集合式 SaaS 后台”重构为“高端 AI 人才搜索工作台”。
2. 保留 HR 和猎头的高频工作效率，不把工作台做成营销页。
3. 让每个页面都有明确主任务：开始搜索、继续项目、审阅候选人、核验证据。
4. 把复杂结果改为“摘要先行，证据按需展开”，降低阅读和判断成本。
5. 建立统一视觉系统，后续功能可以复用，不再每页单独拼样式。

## 非目标

- 不照搬 Apple logo、产品图、导航命名、动画或文案。
- 不做导出功能。
- 不改变现有 API、数据库 schema、搜索/核验核心逻辑。
- 不在第一轮实现英文切换，只保证样式和结构支持后续语言切换。
- 不牺牲关键数据密度；候选池、项目详情仍要适合反复筛选和比较。

## 设计原则

### 1. 焦点优先

每个页面只突出一个主动作：

- 首页：描述要找的人，进入 AI 搜索。
- 总览：继续最重要的招聘项目或开始新研究。
- 搜人：编辑搜索画像并启动研究。
- 项目详情：围绕一个岗位管理候选人、证据和下一轮搜索。
- 候选池：快速审阅、筛选、推进候选人状态。
- 核验台：粘贴候选人材料并生成可审计报告。

### 2. 轻导航

控制台从重型深色侧栏改为“轻顶栏 + 桌面侧向 rail + 移动底部导航”的混合结构。

- 桌面端保留左侧导航，但宽度更窄，使用线性图标和文字，弱化背景重量。
- 顶部提供当前页面标题、主动作和账号入口。
- 移动端保留底部导航，但去掉 emoji，改成一致的图标和短标签。

### 3. 材质感但不过度装饰

使用接近 Apple 的浅色材质：

- 页面背景：`#F5F5F7`。
- 主内容卡：白色或半透明白，细边框，轻阴影。
- 关键舞台区：更大留白、更强层级、更少装饰。
- 不使用大面积紫蓝渐变、装饰 orb、复杂插画或营销式漂浮卡片。

### 4. 渐进展示

AI 研究结果复杂，不能一屏堆完。

- 第一层：结论摘要、候选人数量、证据强度、主要风险。
- 第二层：候选人列表和比较矩阵。
- 第三层：单候选人画像、声称、证据、来源链接和风险。
- 第四层：原始搜索计划、来源执行、补证和反馈优化。

### 5. 可信、安静、专业

SignalHire 的差异是“证据”和“交叉验证”，不是炫技。

- 颜色只服务语义：匹配分、证据质量、风险、状态。
- 动效只用于状态转变和研究进行中感知。
- 文案短，避免大段解释功能。

## 信息架构

### 公开首页

目标：向新用户解释产品，并把用户带入搜索。

结构：

1. 顶部轻导航：品牌、产品能力入口、登录/进入控制台。
2. 首屏 hero：SignalHire 名称 + 一句话价值主张 + 大输入框。
3. 可信来源条：GitHub、Scholar、arXiv、LinkedIn、公司页面、会议、博客等来源类型。
4. 产品产出展示：shortlist、候选人画像、证据审计、项目空间。
5. 对比区：传统简历搜索 vs SignalHire 证据搜索。
6. 示例报告。
7. CTA。

视觉方向：

- 更接近 Apple 首页的强首屏和分段展示。
- Hero 不使用分屏卡片，不做复杂渐变插画。
- 使用产品真实界面截图或简化的产品预览，而不是抽象图形。

### 控制台壳子

目标：降低后台感，让工作台更像高端生产力产品。

桌面布局：

- 左侧窄 rail：Logo、总览、项目、搜人、核验、候选池、历史。
- 顶部 bar：页面标题/面包屑、主 CTA、账号入口。
- 主内容：最大宽度随页面类型变化，不统一锁死 `max-w-5xl`。

移动布局：

- 顶部显示品牌、当前页面、账号动作。
- 底部 tab 保留 4 个核心入口：总览、项目、搜人、候选池。
- 低频入口如历史、设置放入更多菜单。

### 总览

目标：让用户一进来知道“下一步做什么”。

模块：

1. 今日焦点：进行中项目、活跃搜索、待审候选人。
2. 主动作区：新建项目、智能搜人、核验候选人。
3. 项目继续卡：最多展示 3-4 个最重要项目。
4. 进行中研究：显示实时状态和可停止动作。
5. 最近完成：作为次要信息。

交互：

- KPI 不再是首屏主角，改为轻量数字带。
- 空状态直接给“创建第一个项目”和“开始一次搜索”。

### 搜人页

目标：把“生成 AI 人才 shortlist”变成一个清晰的研究流程。

流程：

1. 搜索画像编辑
   - 大输入框置于页面中心。
   - 项目上下文显示为柔和胶囊。
   - “生成搜索计划”与“开始研究”分离。

2. 搜索计划确认
   - 必备条件、加分项、排除项、信息源策略。
   - 以分组面板展示，可编辑。
   - 主按钮：“开始深度研究”。

3. 研究进行中
   - 顶部显示当前动作：正在拆解需求、搜索关键词、读取来源、交叉验证。
   - 中间显示来源流和统计：搜索次数、抓取页面、已覆盖来源组。
   - 保留停止搜索按钮，位置固定且可见。

4. 结果输出
   - 顶部是 shortlist 交付摘要。
   - 接着是候选人卡片网格。
   - 再往下是比较矩阵、证据图、来源覆盖、补证建议。
   - 反馈区放在结果摘要之后，使用选择题。

视觉：

- 搜索输入是页面主舞台，不被小卡片包围。
- 研究进度使用时间线和轻动效，不只显示 spinner。

### 项目列表

目标：把招聘项目变成清晰的工作空间入口。

结构：

- 顶部大标题：招聘项目。
- 筛选使用 segmented control。
- 项目卡片显示：职位名、brief 摘要、候选人数、进行中研究、状态。
- 新建项目弹窗改为更清晰的两步表单：项目名称、招聘画像。

视觉：

- 卡片尺寸稳定，hover 只做轻微抬升。
- 状态用小圆点和文字，不用高饱和彩色块。

### 项目详情

目标：围绕一个岗位完成“继续搜索、审阅、推进、复盘”。

结构：

1. 项目 hero
   - 岗位名、状态、brief 摘要。
   - 主 CTA：在本项目下搜人。
   - 次 CTA：核验候选人、编辑项目。

2. 候选人状态带
   - 总候选人、待联系、已联系、面试中、已 hire、已拒。
   - 可点击筛选。

3. 候选人工作区
   - 左侧候选人列表。
   - 右侧详情面板。
   - 桌面端双栏，移动端点击候选人进入详情。

4. 历史研究
   - 折叠在页面下方。
   - 默认展示最近 5 条。

交互：

- 点击“在本项目下搜人”只带入项目上下文和 brief，不自动启动搜索。
- 候选人状态变化要有即时反馈。

### 候选池

目标：用于集中审阅跨项目候选人。

结构：

- 顶部筛选：全部、状态、项目。
- 候选人列表：姓名、当前角色、匹配分、证据强度、来源数、状态。
- 详情面板：候选人画像、证据、备注、状态推进、AI 外联。

交互：

- 详情面板保持 sticky。
- 移动端使用抽屉或详情页，不在窄屏强行双栏。

### 核验台

目标：将候选人材料变成可信报告。

结构：

- 大输入框：粘贴候选人简介、LinkedIn、简历文本或公开链接。
- 核验进行中：显示正在搜索/读取/对账。
- 报告摘要：总体可信度、红旗、已验证声明、查无实据声明。
- 证据详情：按声明分组。

### 结果组件

当前 `components/result.tsx` 承担大量展示职责。重构时拆成以下展示单元：

- `EvidenceBadge`：证据质量、来源数、裁决。
- `CandidateScore`：匹配分和质量分。
- `CandidateSummaryCard`：候选人摘要卡。
- `ClaimEvidenceList`：声明和证据。
- `SourceCoveragePanel`：来源覆盖。
- `ResearchTimeline`：实时搜索进度。
- `DeliverySummary`：交付摘要。

这些组件要服务搜索结果、项目详情、候选池和公开报告，不绑定单一页面。

## 视觉系统

### 色彩

- 页面背景：`#F5F5F7`
- 主文字：`#1D1D1F`
- 次文字：`#6E6E73`
- 弱文字：`#86868B`
- 卡片背景：`rgba(255,255,255,0.82)` 或 `#FFFFFF`
- 细线：`rgba(0,0,0,0.08)`
- 主动作蓝：`#0071E3`
- 深色强调：`#1D1D1F`
- 成功/验证：emerald 系
- 风险/矛盾：red 系
- 未确认/待核实：amber 系

### 字体

继续使用 Geist Sans 和 Geist Mono。排版接近 Apple 的层级：

- 页面标题：更大、更短、更有留白。
- 工作台正文：14-15px，行高充足。
- 数字：tabular nums。
- 不使用负 letter-spacing。

### 圆角和材质

- 大舞台容器：24px。
- 普通卡片：18-20px。
- 小控件：999px 或 12px。
- 阴影更轻，优先使用边框和背景层级。

### 图标

不再使用 emoji 作为主要图标。

- 使用现有 `react-icons` 或引入统一图标集。
- 图标大小固定，文本不因图标变化抖动。
- 未知或复杂图标需要 hover tooltip。

### 动效

- 页面进入：轻微 fade-up。
- 卡片 hover：1-2px 抬升，阴影轻微增强。
- 搜索进行中：时间线增量出现，当前步骤脉冲。
- 尊重 `prefers-reduced-motion`。

## 交互规则

1. 所有主按钮文案必须是动作，不是状态。
2. 一页只允许一个最高优先级主按钮。
3. 搜索、核验、补证、合并、删除都要有明确 loading/disabled 状态。
4. 错误信息必须告诉用户下一步，而不是只显示 HTTP 状态。
5. 长文本必须可展开或滚动，不能撑破布局。
6. 移动端不允许文字和按钮重叠。
7. 进行中的搜索必须可停止。

## 实施分期

### Phase 1：设计系统和壳子

- 更新全局 CSS token。
- 新增共用 UI primitives。
- 重构控制台 layout。
- 替换 emoji 导航。
- 调整公开首页首屏和导航。

验收：

- `/`、`/app`、`/app/projects` 在桌面和移动端视觉一致。
- 构建通过。
- 无明显文本重叠。

### Phase 2：核心工作流

- 重构 `/app/search` 和 `ResearchTool` 的输入、计划、进度、结果布局。
- 重构 `/app/verify` 的输入和报告布局。
- 抽出 `ResearchTimeline`、`DeliverySummary`、`CandidateSummaryCard`。

验收：

- 搜索过程能实时显示正在搜索和读取的内容。
- 停止搜索按钮始终可见。
- 结果先显示平台语言摘要，再展开原始证据。

### Phase 3：项目与候选池

- 重构项目列表、项目详情、候选池。
- 项目详情形成岗位工作空间。
- 候选人详情改为稳定双栏/移动抽屉。

验收：

- 创建项目后，点击“在本项目下搜人”不会自动开始搜索。
- 候选人状态筛选、备注、外联入口仍可用。

### Phase 4：公开报告与细节打磨

- 重构 `/r/[id]` 公开报告。
- 完成空状态、错误状态、加载状态。
- 做跨断点截图检查。

验收：

- 公开报告可读性提升，证据和风险清晰。
- 桌面和移动端截图无布局问题。

## 测试与验证

每个实施 PR 至少执行：

- `node --test` 相关轻量测试。
- `npm run build` in `web`。
- 对核心页面进行浏览器截图检查：`/`、`/app`、`/app/search`、`/app/projects`、`/app/shortlist`。
- 移动宽度检查：375px 或接近移动端视口。
- 检查 console error。

视觉验收标准：

- 不再出现 emoji 主导航。
- 页面背景、卡片、按钮、筛选控件使用统一 token。
- 每个核心页面有明确主动作。
- 搜索进行中有可见实时研究内容。
- 结果阅读顺序是摘要 → shortlist → 证据详情。

## 风险

1. 重构范围大，容易引入页面功能回归。
   - 应对：按 phase 拆 PR，每个 PR 只改一组页面。

2. Apple 风格可能让信息密度过低。
   - 应对：营销页使用大留白；工作台保留表格、列表和双栏，但降低视觉噪音。

3. `ResearchTool` 文件较大，直接重构风险高。
   - 应对：先抽展示组件，再逐步替换 JSX，不改请求和状态机。

4. 移动端复杂详情页容易拥挤。
   - 应对：移动端使用单列和抽屉，不强行复用桌面双栏。

## 决策

采用“Apple 风格的产品工作台重构”方案：迁移 Apple 的焦点、留白、材质、层级和渐进展示原则，但不复制其消费电子官网结构。SignalHire 仍然是 HR/猎头的高频工作工具，核心体验是快速搜索、审阅、验证和推进候选人。
