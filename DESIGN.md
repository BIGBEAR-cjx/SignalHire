## 一句话

输入一句自然语言招聘需求 → 用 MiroMind 深度搜索全网找到候选工程师 →
**对每个候选人的"声称"做跨源交叉验证**，标出哪些经历有实证、哪些查无实据、哪些自相矛盾。

竞品 juicebox/lessie 给你 500 个简历。我们给你 5 个**已经被核实过**的人。
这就是 Deep Research Track 的杀手锏。

**产品两个模式(Day 2 后确定)：**
1. **搜人**(`engine.mjs`)：一句招聘需求 → 3 个不同真人候选 + 每人交叉验证声称。慢(~4-8 分钟)。
2. **验证候选人 / 打脸**(`verify.mjs`)：粘一份候选人自述 → 可信度报告(verified/contradicted/unverified + 红旗)。快(~2 分钟)，**是 demo 头牌** —— 现场粘一份注水简历，看它当场拆穿。

## Problem Statement

招聘里最贵的不是"找到人"，是"核实人"。简历和 LinkedIn 上的自我描述经常注水
("核心贡献者""5 年大厂经验")，招聘方要花大量时间去 GitHub、论文、博客交叉比对。
现有 AI 猎头工具(juicebox/lessie)强在**广度**(搜得多)，弱在**可信度**(不验证)。

## What Makes This Cool (评委会记住的"哇"时刻)

演示中搜出一个候选人，系统说：
> "声称：tokio 核心贡献者 → ✅ 已验证：tokio-rs/tokio 有 47 个已合并 PR"
> "声称：曾在 Google 工作 5 年 → ⚠️ 查无实据：找不到任何公开证据"

每条结论都附**可点击的证据链接**。这把"AI 又在编"变成"AI 拿出了证据"。

## Constraints (硬约束，不可谈判)

- **时间**：2026-05-21 → 2026-06-01，每天约 2 小时，总计约 20 小时。
- **开发者**：编程小白，只用 Claude Code vibe coding。
- **裁掉一切非核心**：无登录、无数据库、无多用户、无设置页。一个能跑通的演示流程就是全部目标。

## Premises (已确认)

1. Deep Research Track 奖励**深度与验证**，不是广度。演示核心是交叉验证，不是返回多少人。
2. 评委的怀疑才是真正的对手。说"验证过"就必须**亮出证据**(真实 PR、真实链接)。
3. 黑客松的"交付"= 一个在线 URL + 一段 2-3 分钟录屏 demo。
4. 20 小时 + 小白 = 一切不在"单一可演示流程"路径上的东西全部砍掉。

## 技术栈 (Day 1 实测后已定)

> **架构调整**：实测发现 MiroMind 自己就是完整 deep research agent，会自己上网搜+交叉核实。
> 所以**不需要 dzhng/deep-research 的搜索循环，也不需要 Firecrawl**。工作量大减。

| 部分 | 选型 | 说明 |
|------|------|------|
| 引擎 | **MiroMind API** (`mirothinker-1-7-deepresearch-mini`) | 自己上网搜+交叉验证，直接当大脑。OpenAI 兼容，已验证可用 |
| 网页搜索/抓取 | 不需要 (MiroMind 自带) | 实测一次查询自动 55 次搜索 + 8 次抓取，省掉 Firecrawl 依赖 |
| 前端 | Next.js (单页) | 搜索框 + 候选人卡片 + 实时研究流 |
| 部署 | Vercel | 一下午搞定，免费 |
| 参考 | `deep-research/` (已克隆本地) | 仅参考其 prompt / api.ts 结构，不用其搜索循环 |

## 系统流程

```
自然语言需求
  └─> 一个 prompt 交给 MiroMind，它自己:
        · 全网搜候选人 (GitHub / LinkedIn / 博客 / 论文)
        · 逐条提取"声称" → 跨多源找佐证
        · 标 ✅已验证 / ⚠️查无实据 / ❌矛盾 + 证据 URL
  └─> 返回 JSON (choices[0].message.content)
  └─> (可选) 流式把 web_search / fetch_url 步骤实时显示给前端
前端渲染候选人卡片 + 验证面板 + 实时研究流
```

响应结构、延迟数据、运行方式见 `NOTES.md`。引擎雏形见 `engine.mjs`。

## Approaches Considered

- **A (选定)**：搜人 + 验证 web 应用。完整愿景的收敛版，演示最像产品。
- **B**：CLI 优先、UI 最后。作为 A 的执行纪律采纳(先把引擎跑通)。
- **C**：只验证一个人的"信任报告"。作为时间不够时的**逃生口**。

## Recommended Approach

**A，但用 B 的纪律执行**：先把引擎(`engine.mjs`)跑通(已起步)，再包 UI。
如果到 Day 5 还没把验证流可视化跑顺，**立刻砍成 C**(只深度验证一个人)，照样能交亮眼 demo。

## 头号风险 + 兜底方案

API 接入已验证 ✅ (OpenAI 兼容、key 可用、能自主全网搜索)。**新的头号风险是延迟**：
深度查询要 5 分钟+ (实测 4'45" 还没答完)。

**兜底**：
1. **预缓存** hero 查询 → 现场回放，同时作 API 挂掉的兜底。
2. **流式显示**研究过程 → 把等待变成表演(评委看着它全网找证据)。
3. **prompt 限范围**(3 人、每人 2-3 条声称) → 缩短时间。
4. 若输出 JSON 不干净 → 加一步便宜模型做格式化整理。

## Success Criteria

- [ ] 一个在线 URL，输入一句招聘需求，返回 ≥3 个候选工程师卡片。
- [ ] 每个候选人有 ≥1 条被**交叉验证**的声称，带可点击证据链接。
- [ ] demo 里至少出现一次"声称 vs 实际不符"的打脸时刻。
- [ ] MiroMind 是验证/搜索的核心，故事讲得通。
- [ ] 2-3 分钟录屏 + 提交文案完成。

## Distribution Plan

- 在线 demo：Vercel 部署，免费域名。
- 录屏 demo：2-3 分钟，重点演交叉验证的打脸瞬间。
- 代码：GitHub 公开 repo + 清晰 README(怎么跑、用了 MiroMind 什么能力)。
